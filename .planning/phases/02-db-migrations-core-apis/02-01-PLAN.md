# Plan 2-01: DB Schema, Migrations, Core APIs & Kafka Integrations

## Objective
Build and configure the complete backend core of the application for Phase 2. This includes establishing Flyway-driven database migrations for PostgreSQL, implementing stateless Spring Security with custom JWT authentication, writing core catalog CRUD and checkout flows (using database pessimistic write locks), setting up sequential Kafka-driven bulk CSV imports, and configuring Kafka-driven asynchronous audit logging.

## Code Patterns (Model Resilience)
Reference these existing files for code style:
- [ProductController.java](file:///c:/Users/leocg/Documents/GitHub/challenge-ecommerce-gila/src/main/java/com/gila/ecommerce/controller/ProductController.java) — matches REST interface mapping rules.
- [ECommerceApplication.java](file:///c:/Users/leocg/Documents/GitHub/challenge-ecommerce-gila/src/main/java/com/gila/ecommerce/ECommerceApplication.java) — matches package conventions.

Conventions:
- **Class Line Limits:** Under no circumstances should any class exceed 600 lines. Heavy processes (e.g. CSV import processing, security configurations) must be decomposed into multiple specialized classes.
- **Method-Level Comments Only:** Place comments *only* on method signatures (JavaDoc/TSDoc style). Absolutely no comments are allowed inside method/function bodies.
- **SQL & JPA Security:** Parameterize all queries. Restrict DB interactions to JPA repository interfaces to prevent SQL injection.
- **Data Types:** Use `BigDecimal` inside entities and calculations to represent prices, and convert to `Double` inside DTO structures. Use `UUID` for identifiers.
